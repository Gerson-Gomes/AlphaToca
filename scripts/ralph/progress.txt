## Codebase Patterns
- Seed IDs: random seeds use `crypto.randomUUID()` (not `faker.string.uuid()`); stable demo IDs live in `prisma/demoIds.ts` as canonical UUIDv4 constants — don't replace those constants with `randomUUID()` calls because they're referenced by tests and Firebase UID mappings.
- UUID validation: routes use `z.string().uuid()`; any seed id you introduce must match `/^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i`.
- Test runner: `npx vitest run` (vitest config in `vitest.config.ts`). Many pre-existing tests fail due to an unfinished Auth0→Firebase migration (`userService.upsertUserFromAuth0` missing, `tests/auth.test.ts`, `tests/ingestKnowledge.test.ts`) — treat those as baseline noise, not regressions.
- Grep scope for "seed" lookups: include `prisma/` (generators.ts, demoData.ts, demoIds.ts, seed.ts) — seeds are not under `src/` in this repo.
- Typecheck: `npx tsc --noEmit` — must pass before commit.
- `.env` DATABASE_URL points at the live Supabase project — DO NOT run `prisma migrate dev/deploy/reset` locally; hand-author `prisma/migrations/<stamp>_<name>/migration.sql` instead and verify with `npx prisma validate` + `npx prisma generate`. Reviewer runs the actual `migrate reset --force` on a disposable dev DB before landing.
- Swagger is split: `src/config/swagger.ts` owns component schemas (e.g., `Property`); per-route JSDoc blocks in `src/routes/*.ts` own request/response shapes — when an enum changes, both files need an edit.
- Prisma enum value rename: use `ALTER TYPE "X" RENAME VALUE 'OLD' TO 'NEW';` — it rewrites the catalog label in place, so existing rows carry over with zero data loss (no `UPDATE` pass required).
- `propertyValidation.ts` uses `z.nativeEnum(PropertyStatus)` — it re-reads the Prisma client enum at import time, so bumping the enum only requires a `prisma generate`, not a Zod edit.
