## Codebase Patterns
(Carry over from archive/2026-05-07-landlord-backend-closeout/progress.txt ŌĆö add new ones as discovered during this run.)

- **Auth middleware mock**: tests that exercise 401/403 use the `importOriginal` pattern to partially mock `authMiddleware` (keeps `requireRole` real, swaps `checkJwt`/`authSyncMiddleware` with test doubles that inject a fake `req.localUser`). Apply when writing new auth-guarded tests.
- **Hand-author migrations**: NEVER run `prisma migrate dev`. Write `prisma/migrations/<stamp>_<slug>/migration.sql` by hand, then verify byte-for-byte with `npx prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script`.
- **Decimal deserialization**: Prisma `@db.Decimal(10,2)` columns come back as `Prisma.Decimal` ŌĆö convert to plain number with `Number(row.field)` before putting them in HTTP response bodies.
- **PropertyImage.url format is inconsistent by origin**: `propertyImageStorageService.savePropertyImages` writes RELATIVE URLs (`/uploads/<propertyId>/<uuid>.png`), but `prisma/demoData.ts` seeds ABSOLUTE URLs (`http://10.0.2.2:3000/uploads/<propertyId>/NNNN.png`). Frontend `absoluteImageUrl()` + `_normalizeLocalHost()` handle both. Keep it in mind when adding new image endpoints.
- **CORS + static ordering in src/app.ts**: `app.use(cors())` is at line 40, BEFORE `app.use('/uploads', express.static(...))` at line 50. `/uploads/*` responses include `Access-Control-Allow-Origin: *`. Do NOT move static before cors.
- **Pre-existing test failures (baseline)**: On `ralph/backend-gaps-finais`, `npx vitest run` reports **23 failing / 618 passing / 65 files** with a CLEAN working tree. The 6 failing files are: `tests/auth.test.ts`, `tests/ingestKnowledge.test.ts`, `tests/ragChainService.test.ts`, `tests/ragRetrieverService.test.ts`, `tests/userSync.test.ts`, `tests/visitValidation.test.ts`. These pre-date this rollout (RAG/ingest/auth plumbing drift) and are unrelated to `/uploads`, CORS, or the P4 user stories. Future stories should treat this count as the baseline ŌĆö a story is non-regressive if it leaves 23 or fewer failures in these same 6 files.
- **Support ticket shape split (admin vs user view)**: `supportTicketService` now exposes two list shapes ŌĆö `SupportTicketAdminView` (with `user`, `assignedTo`, `resolution`, `updatedAt`) for `GET /admin/support/tickets`, and `SupportTicketUserView` (just `id`, `code`, `title`, `description`, `createdAt`, `status`) for `GET /support/tickets`. Keep them separate: the user view intentionally hides triage metadata from the ticket's own opener. When adding new fields, decide which view(s) need them and update the relevant `select:` or `toAdminView` builder ŌĆö don't merge the two.
- **Support routes mount**: `src/routes/supportRoutes.ts` is mounted once via `app.use('/api', authStack, supportRoutes)` in src/app.ts. Every route defined in this file (including `/support/tickets`, `/admin/support/tickets`) is therefore behind `authStack` (checkJwt + authSyncMiddleware). Admin-only routes layer `requireRole('ADMIN')` inside the router itself. When adding a new support endpoint, do NOT re-mount the router in app.ts ŌĆö add the route in supportRoutes.ts and (if it needs admin) gate with `adminOnly`.

### So what IS breaking the placeholder on the frontend?

Out of scope for this US (backend only), but documenting so US-002 doesn't spin its wheels:

1. **`_normalizeLocalHost` is too aggressive for Android emulator**. Seed rows store `http://10.0.2.2:3000/uploads/...`, which IS the correct host alias when the Flutter app runs on an Android emulator. The frontend's `_normalizeLocalHost` (per GAPS_FINAIS.md ┬¦1.4) rewrites `10.0.2.2` ŌåÆ `localhost`, which is valid for Flutter web but breaks on the Android emulator (where `localhost` points to the emulator itself, not the host machine). Recommendation: make the normalizer platform-aware (keep `10.0.2.2` on Android, rewrite to `localhost` only on web/desktop/iOS).

2. **URL format inconsistency in DB**. `savePropertyImages()` writes RELATIVE URLs (`/uploads/<propertyId>/<uuid>.png`); seeds write ABSOLUTE URLs (`http://10.0.2.2:3000/uploads/<propertyId>/0001.png`). Both work today thanks to the `absoluteImageUrl` helper, but aligning on relative-only (and letting the client prepend its configured backend host) would remove an entire class of host-resolution bugs. Tracked separately ŌĆö do not do it as part of US-002.

### US-002 decision

Given that the live curl proves 200 / correct CORS / correct Content-Type, US-002 has **no backend change to apply**. The recommended next step is to mark US-002 as a no-op with this investigation linked, or reframe it as a frontend fix (platform-aware `_normalizeLocalHost`) in the Flutter repo. Opening a note in the PRD `notes` field on US-002 rather than making cosmetic backend edits that would risk regressions.
