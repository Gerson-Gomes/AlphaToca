## Codebase Patterns
(Carry over from archive/2026-05-07-landlord-backend-closeout/progress.txt — add new ones as discovered during this run.)

- **Auth middleware mock**: tests that exercise 401/403 use the `importOriginal` pattern to partially mock `authMiddleware` (keeps `requireRole` real, swaps `checkJwt`/`authSyncMiddleware` with test doubles that inject a fake `req.localUser`). Apply when writing new auth-guarded tests.
- **Hand-author migrations**: NEVER run `prisma migrate dev`. Write `prisma/migrations/<stamp>_<slug>/migration.sql` by hand, then verify byte-for-byte with `npx prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script`.
- **Decimal deserialization**: Prisma `@db.Decimal(10,2)` columns come back as `Prisma.Decimal` — convert to plain number with `Number(row.field)` before putting them in HTTP response bodies.
- **PropertyImage.url format is inconsistent by origin**: `propertyImageStorageService.savePropertyImages` writes RELATIVE URLs (`/uploads/<propertyId>/<uuid>.png`), but `prisma/demoData.ts` seeds ABSOLUTE URLs (`http://10.0.2.2:3000/uploads/<propertyId>/NNNN.png`). Frontend `absoluteImageUrl()` + `_normalizeLocalHost()` handle both. Keep it in mind when adding new image endpoints.
- **CORS + static ordering in src/app.ts**: `app.use(cors())` is at line 40, BEFORE `app.use('/uploads', express.static(...))` at line 50. `/uploads/*` responses include `Access-Control-Allow-Origin: *`. Do NOT move static before cors.
- **Pre-existing test failures (baseline)**: On `ralph/backend-gaps-finais`, `npx vitest run` reports **23 failing / 618 passing / 65 files** with a CLEAN working tree. The 6 failing files are: `tests/auth.test.ts`, `tests/ingestKnowledge.test.ts`, `tests/ragChainService.test.ts`, `tests/ragRetrieverService.test.ts`, `tests/userSync.test.ts`, `tests/visitValidation.test.ts`. These pre-date this rollout (RAG/ingest/auth plumbing drift) and are unrelated to `/uploads`, CORS, or the P4 user stories. Future stories should treat this count as the baseline — a story is non-regressive if it leaves 23 or fewer failures in these same 6 files.
- **Support ticket shape split (admin vs user view)**: `supportTicketService` now exposes two list shapes — `SupportTicketAdminView` (with `user`, `assignedTo`, `resolution`, `updatedAt`) for `GET /admin/support/tickets`, and `SupportTicketUserView` (just `id`, `code`, `title`, `description`, `createdAt`, `status`) for `GET /support/tickets`. Keep them separate: the user view intentionally hides triage metadata from the ticket's own opener. When adding new fields, decide which view(s) need them and update the relevant `select:` or `toAdminView` builder — don't merge the two.
- **Support routes mount**: `src/routes/supportRoutes.ts` is mounted once via `app.use('/api', authStack, supportRoutes)` in src/app.ts. Every route defined in this file (including `/support/tickets`, `/admin/support/tickets`) is therefore behind `authStack` (checkJwt + authSyncMiddleware). Admin-only routes layer `requireRole('ADMIN')` inside the router itself. When adding a new support endpoint, do NOT re-mount the router in app.ts — add the route in supportRoutes.ts and (if it needs admin) gate with `adminOnly`.
- **Check whether a P4 endpoint was already built in an earlier LL-rollout before writing code**. LL-011→LL-015 delivered the full conversations/messages API (GET /conversations, GET /conversations/resolve, GET /conversations/:id/messages, POST /conversations/:id/messages, POST /conversations/:id/read). US-004→US-006 in this PRD overlap word-for-word with those endpoints. Always grep `src/routes/` + `src/controllers/` for the path before starting the story — if it exists, run `git log --oneline -- <file>` to see which archived rollout produced it. If shape + guards + tests already match the AC, close the story as a no-op in the PRD `notes` and skip the code churn. Do NOT rewrite equivalent logic.
- **Conversation schema uses landlordId/tenantId columns, not a `participants[]` array**. When an AC asks to "exclude req.localUser.id from participants", translate that to `WHERE landlordId = userId OR tenantId = userId` + `counterpart = (row.landlordId === userId ? row.tenant : row.landlord)`. The schema is intentionally denormalized this way (single UNIQUE index on `(propertyId, landlordId, tenantId)` anchors the "one 1:1 thread per property per pair" invariant). Do NOT add a participants array — it would duplicate state and break the upsert contract in `conversationService.resolve`.

### So what IS breaking the placeholder on the frontend?

Out of scope for this US (backend only), but documenting so US-002 doesn't spin its wheels:

1. **`_normalizeLocalHost` is too aggressive for Android emulator**. Seed rows store `http://10.0.2.2:3000/uploads/...`, which IS the correct host alias when the Flutter app runs on an Android emulator. The frontend's `_normalizeLocalHost` (per GAPS_FINAIS.md §1.4) rewrites `10.0.2.2` → `localhost`, which is valid for Flutter web but breaks on the Android emulator (where `localhost` points to the emulator itself, not the host machine). Recommendation: make the normalizer platform-aware (keep `10.0.2.2` on Android, rewrite to `localhost` only on web/desktop/iOS).

2. **URL format inconsistency in DB**. `savePropertyImages()` writes RELATIVE URLs (`/uploads/<propertyId>/<uuid>.png`); seeds write ABSOLUTE URLs (`http://10.0.2.2:3000/uploads/<propertyId>/0001.png`). Both work today thanks to the `absoluteImageUrl` helper, but aligning on relative-only (and letting the client prepend its configured backend host) would remove an entire class of host-resolution bugs. Tracked separately — do not do it as part of US-002.

### US-002 decision

Given that the live curl proves 200 / correct CORS / correct Content-Type, US-002 has **no backend change to apply**. The recommended next step is to mark US-002 as a no-op with this investigation linked, or reframe it as a frontend fix (platform-aware `_normalizeLocalHost`) in the Flutter repo. Opening a note in the PRD `notes` field on US-002 rather than making cosmetic backend edits that would risk regressions.

### Learnings for future iterations
- **Always grep the existing routes/controllers/services before writing new ones when the AC overlaps with an archived rollout**. LL-011→LL-015 produced the full conversations API; US-004→US-006 in this PRD are restatements of the same contract. The pattern observed in US-002 (no-op closure grounded in evidence) applies to US-005 and US-006 too — those will likely be similarly closed by the existing conversationRoutes.ts endpoints. Check before writing.
- **Filename deltas in ACs aren't worth honoring if content matches**. The AC asked for `tests/conversationsList.test.ts` (plural); the actual file is `conversationList.test.ts` (singular). Renaming would break `git log --follow` on the file and produce no behavioral change — accept the delta and document it in the notes. The bar for renaming test files in a tracked commit is: the new name reveals something the old name hid, OR a test-runner regex requires it.
- **Schema uses denormalized landlordId/tenantId, not participants[]**. Do NOT try to add a participants array to satisfy an AC wording — it would duplicate state and break `conversationService.resolve`'s UNIQUE(propertyId, landlordId, tenantId) upsert contract. Translate "participants[]" in ACs to the landlordId/tenantId pair directly; the counterpart is `(row.landlordId === userId ? row.tenant : row.landlord)`.
- **`Conversation.lastMessageAt` is an explicit column that DOES NOT yet exist** — it's scheduled for US-006. The service currently derives last-message-time from `messages[0].createdAt || conversation.createdAt` via the included `messages` relation. When US-006 lands, it should (a) hand-author a migration adding the column with a backfill UPDATE, (b) update `createMessage` to set `lastMessageAt` in the same tx (possibly via Prisma `update` + nested `messages.create`), (c) swap the service's in-memory sort for an index-backed `orderBy: { lastMessageAt: 'desc' }` in the initial findMany. US-004 is NOT blocked on this — the fallback is semantically equivalent.
- **The `the-other` user in conversationList.test.ts is the AC's "user C"**. The test mocks `mockList.mockResolvedValue([])` for OTHER_ID, asserting the route forwards the correct userId without returning other users' data. Combined with the service-level assertion that `where` is scoped to the user's id, the "user C isolation" AC is covered end-to-end without needing a dedicated `three_users_cross_leak` test.
